---
title: "Scanning a cluster"
description: "Scan a running Kubernetes cluster for misconfigurations using Kubescape."
---

Running `kubescape scan` against a live cluster checks it against Kubescape's built-in security controls and reports findings grouped by category.

## Basic scan

```bash
kubescape scan
```

This connects to the cluster in your current kubeconfig context and runs the default set of frameworks (`clusterscan`, `mitre`, `nsa`).

### Understanding the output sections

<AccordionGroup>
  <Accordion title="Control plane">
    Checks the configuration of Kubernetes control plane components — the API server, etcd, and their settings. Results are shown as pass (✅) or fail (❌) because control plane checks are binary: either the setting is correct or it is not. Each row links to a documentation page for that control.
  </Accordion>
  <Accordion title="Access control">
    Lists controls that evaluate RBAC rules and permissions. The **Resources** column shows how many Kubernetes objects failed the check. A count of `0` means all evaluated resources passed. Run the suggested `kubescape scan control <ID> -v` command to see exactly which objects failed.
  </Accordion>
  <Accordion title="Secrets">
    Checks for credentials or sensitive data that may be exposed in configuration files, environment variables, or other accessible locations.
  </Accordion>
  <Accordion title="Network">
    Reports workloads that lack network policies, which limits your ability to restrict traffic between pods.
  </Accordion>
  <Accordion title="Workload">
    Evaluates pod and container security settings across your cluster — privilege escalation, host namespace sharing, and similar runtime security concerns.
  </Accordion>
  <Accordion title="Highest-stake workloads">
    Kubescape ranks workloads by their estimated blast radius if exploited. Use the provided `kubescape scan workload` commands to inspect these in detail.
  </Accordion>
  <Accordion title="Compliance score">
    The percentage of controls passing for each framework. This is an average of per-control compliance scores across all evaluated resources. A higher score means fewer failed controls.
  </Accordion>
</AccordionGroup>

## Selecting a cluster context

<CodeGroup>

```bash Alternative kubeconfig file
kubescape scan --kubeconfig /path/to/cluster.conf
```

```bash Different kube-context
kubescape scan --kube-context staging-cluster
```

</CodeGroup>

<Note>
  `--kubeconfig` is a global flag inherited from kubectl. `--kube-context` selects a named context within the kubeconfig file.
</Note>

## Filtering namespaces

```bash
# Scan only specific namespaces
kubescape scan --include-namespaces production,staging

# Exclude namespaces from the scan
kubescape scan --exclude-namespaces kube-system,kube-public
```

<Warning>
  When `--exclude-namespaces` is set, Kubescape does not scan cluster-scoped objects. If you need a complete picture of cluster-level resources, run without this flag.
</Warning>

## Scanning a specific workload

After a cluster scan, Kubescape highlights high-stake workloads. You can drill into any workload by kind and name:

```bash
kubescape scan workload <Kind>/<name> --namespace <namespace>
```

Examples:

```bash
kubescape scan workload DaemonSet/gadget --namespace gadget
kubescape scan workload Deployment/nginx --namespace default
kubescape scan workload Pod/my-cluster-kafka-0 --namespace kafka
```

This command fetches the live resource from the cluster and evaluates it against all applicable controls, including vulnerability data if image scanning is enabled.

## Failing on thresholds

Use thresholds to enforce a minimum security bar in CI/CD pipelines or automated checks. Kubescape exits with code `1` when a threshold is breached.

### Compliance threshold

Fail if the overall compliance score falls below a percentage:

```bash
kubescape scan --compliance-threshold 80
```

You can combine this with a specific framework:

```bash
kubescape scan framework nsa --compliance-threshold 80
```

### Severity threshold

Fail if any controls at or above a given severity level fail:

```bash
kubescape scan --severity-threshold high
```

Valid values: `low`, `medium`, `high`, `critical`.

<Tip>
  Combine both thresholds in CI to require a minimum compliance score **and** zero high-or-critical failures:

  ```bash
  kubescape scan --compliance-threshold 80 --severity-threshold high
  ```
</Tip>

## Verbose output

By default, Kubescape only shows failed resources. Add `-v` to display all evaluated resources:

```bash
kubescape scan -v
```

## Display modes

The `--view` flag controls how results are grouped:

| Value | Description |
|-------|-------------|
| `security` | Default — grouped by security category (control plane, access control, etc.) |
| `control` | Grouped by individual control |
| `resource` | Grouped by Kubernetes resource |

```bash
kubescape scan --view resource
```
